## 核心觀念：什麼是 Harness？

**Harness（馬具／線束）** 是包在 AI 模型外面、把原始認知能力轉化為可靠產出的整套系統。

| 類比 | 模型 | Harness |
|------|------|---------|
| 馬 | 馬的力量（原始能力） | 韁繩、馬鞍、胸帶 |
| 電腦 | CPU | 作業系統 |
| AI Agent | LLM（推理、生成） | 工具、驗證、迴圈、記憶、護欄 |

**關鍵洞察**：2024 年之前模型是瓶頸，再好的 harness 也救不了一個連基本邏輯都搞不定的模型。2025 年 GPT-4.1、Claude Sonnet 4、Gemini 2.5 Pro 把基本能力拉到「够用」水準，於是大家發現：**模型差距越來越小，但同一個模型在不同 harness 下的表現天差地遠**。

```
┌─────────────────────────────────────────┐
│        Harness Engineering (2026)       │
│  ┌───────────────────────────────────┐  │
│  │      Context Engineering (2025)    │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │    Prompt Engineering        │  │  │
│  │  │  (2022-2024)                 │  │  │
│  │  └─────────────────────────────┘  │  │
│  │  + 動態上下文、RAG、對話歷史      │  │

│  + 工作流、護欄、反饋、生命周期管理     │
└─────────────────────────────────────────┘
```

| 階段 | 關注點 | 類比 |
|------|--------|------|
| Prompt Engineering | 寫出完美的指令 | 給承包商寫一封措辭完美的郵件 |
| Context Engineering | 建構正確的資訊環境 | 整個簡報包——組織圖、文件、過往溝通 |
| Harness Engineering | 設計整個運行環境 | 設計整個辦公室——歸檔系統、審批流程、安全制度 |

## Harness 的六層結構

| 層次 | 名稱 | 說明 |
|------|------|------|
| 1 | 迴圈（Loop） | 觀察 → 決定 → 行動 → 驗證 → 更新，不斷循環直到完成 |
| 2 | 工具（Tools） | 讓 AI 不只能說話，還能讀檔案、跑程式、call API、搜尋網頁 |
| 3 | 上下文（Context） | 控制 AI 看到哪些資訊、不看哪些，避免資訊過載 |
| 4 | 持久化（Persistence） | 跨對話、跨執行記住之前做過什麼 |
| 5 | 驗證（Verification） | 做完之後跑測試、檢查語法、自我審查，確保品質 |
| 6 | 限制（Constraints） | 哪些事不准做、哪些檔案不能碰、花費上限多少 |

## 兩大核心組件：引導器與傳感器

### 引導器（Guides）—— 前饋控制

在生成之前引導 Agent 行為，提高一次性做對的機率。

| 類型 | 計算/推理 | 實例 |
|------|----------|------|
| 架構文檔 | 推理型 | CLAUDE.md 描述倉庫結構和約定 |
| 編碼規範 | 計算型 | ESLint/Prettier 配置約束輸出格式 |
| 啟動腳本 | 計算型 | 專案初始化腳本創建正確的檔案結構 |
| 類型定義 | 計算型 | TypeScript 類型收窄解題空間 |
| 範例模式 | 推理型 | 展示「正確做法」的程式碼片段 |
| Code mods | 計算型 | 有訪問 OpenRewrite recipes 的工具 |

**實證經驗**：CLAUDE.md 控制在 60 行以內效果最好。LLM 自動生成的冗長文件反而降低 20% 的表現。

### 傳感器（Sensors）—— 反饋控制

在生成之後幫助 Agent 自我糾錯。

| 類型 | 計算/推理 | 能發現什麼 |
|------|----------|-----------|
| 單元測試 | 計算型 | 行為正確性 |
| 類型檢查 | 計算型 | 類型錯誤、介面不匹配 |
| 程式碼檢查 | 計算型 | 風格違規、常見 bug |
| AI 程式碼審查 | 推理型 | 語義問題、設計缺陷 |
| 整合測試 | 計算型 | 跨組件故障 |
| 結構測試 | 計算型 | 架構邊界違規（如 ArchUnit）|

### 兩者缺一不可

- **只有引導器** → Agent 按模式做但從不驗證 → 靜默失敗
- **只有傳感器** → Agent 犯錯→發現→修復→再犯同樣的錯 → 緩慢低效
- **引導器 + 傳感器** → Agent 通常一次做對，偶爾出錯能自動修復 → 可靠

### 計算型 vs 推理型

| 類型 | 特性 | 速度 | 可靠性 |
|------|------|------|--------|
| **計算型** | 測試、linter、類型檢查、靜態分析 | 毫秒到秒 | 確定性，非常可靠 |
| **推理型** | AI 程式碼審查、LLM as judge | 較慢，GPU/NPU 占用 | 非確定性，但语义理解更強 |

**關鍵經驗**：傳感器的輸出要為 LLM 優化。例如 lint 錯誤信息寫成 `Error: unused variable 'x' on line 42` 就比一個錯誤碼有用得多。格式化成糾錯指令效果最好。

## Martin Fowler 的三大 Harness 類別

Martin Fowler 將 Harness 分為三個 regulation categories：

### 1. Maintainability Harness（可維護性 Harness）

調節內部代碼品質和可維護性。這是目前最成熟的類型，因為有大量現成工具可用。

| 傳感器類型 | 能catch到的問題 |
|-----------|----------------|
| 計算型傳感器 | 重複代碼、循環複雜度、測試覆蓋缺口、架構漂移、風格違規 |
| 推理型傳感器 | 語義重複代碼、冗余測試、暴力修復、過度工程（昂貴且概率性） |

**無法可靠catch的問題**：對問題的誤判、過度工程和不必要功能、誤解指令。

### 2. Architecture Fitness Harness（架構適應性 Harness）

定義和檢查應用程式架構特性的引導器和傳感器。本質上是 Fitness Functions（適應度函數）。

**範例**：
- 引導器：描述效能需求的 Skills + 效能測試反饋是否達標
- 引導器：描述更好的可觀測性編碼約定（如日誌標準）+ 調試指令讓 Agent 反思日誌質量

### 3. Behavior Harness（行為 Harness）

這是最大挑戰——如何引導和感知應用程式功能行為是否符合需求。

**目前常見做法**：
- 前饋：功能規格（從短提示到多檔案描述）
- 反饋：檢查 AI 生成的測試套件是否綠燈、覆蓋率是否合理

**問題**：對 AI 生成測試的質量信心不足。有些人使用 Approved Fixtures 模式來選擇性應對。

## 真實案例

### OpenAI Codex：100 萬行代碼，零手寫

OpenAI Codex 團隊在五個月內生產了超過 100 萬行代碼和 1,500 個 PR，**沒有一行是人類工程師寫的**。

關鍵做法：
- 分層架構通過自定義 lint 規則強制執行
- 結構性測試驗證模組邊界
- 定期「垃圾回收」掃描架構漂移
- 倉庫作為唯一真相源——約束寫在代碼裡，不是指令裡
- 七名工程師全力投入 Harness Engineering
- 平均每位工程師每天 3.5 個 PR，隨團隊成長到七人，吞吐量反而增加

**核心原則**：把約束編碼進系統，而不是寫進提示詞。一條攔截循環引用的 lint 規則比一句「不要創建循環引用」的提示詞可靠一萬倍。

**最大挑戰**：Martin Fowler 引用 OpenAI 結論：「我們現在最大的挑戰集中在設計環境、反饋迴圈和控制系統。」

### LangChain：不換模型，從第 30 名升到第 5 名

在 TerminalBench 2.0 上從 52.8% 跳到 66.5%，模型一點沒換。只改了 Harness：
- 加了自我驗證迴圈
- 注入環境上下文
- 防迷路機制（偵測 Agent 是否在重複編輯同一個檔案）
- 調整推理預算——規劃和驗證時給多一點思考額度，實作時少一點
- 失敗分析——自動分析每次失敗的模式，歸納改進方向

### Vercel：工具從 15 個砍到 2 個

結果：
- 準確率：80% → 100%
- Token 用量：少了 37%
- 速度：快了 3.5 倍

**少即是多。限制本身就是 Harness 的一部分。**

### Epsilla 實驗

同一個模型、同一組提示詞，僅修改運行環境（工具介面、驗證迴圈、約束執行），Agent 表現從 42% 提升到 78%。

### Stripe Minions：每週 1,300 個 PR

Stripe 的自主 Agent 系統：
- "Blueprint" 編排，確定性節點和 Agent 節點分離
- Push 前的 Hook 自動選擇相關 lint 規則
- 兩次失敗升級規則：同一個問題失敗兩次就轉交人類，不再無限重試

## Agent = Model + Harness

| 維度 | Model | Harness |
|------|-------|---------|
| 類比 | CPU | 作業系統 |
| 做什么 | 思考、生成、推理 | 管理、約束、驗證 |
| 怎麼提升 | 等下一代模型 | 今天就能工程化優化 |
| 差異化 | 同質化（效能差距很小） | 獨特的（你的競爭優勢） |

## Ashby 定律

> 調節器的複雜度必須不小於被調節系統。

LLM 能生成幾乎任何東西——解題空間是無限的。為無限空間構建完備的 Harness 是不可能的。但如果承諾固定模式——「所有 API 端點遵循這個結構」——就把解題空間縮小到 Harness 能全面覆蓋的範圍。

**約束不是限制。約束是可靠自主運行的前提條件。**

## 五大核心支柱（Harness Engineering 實務）

### 1. 工具編排（Tool Orchestration）

定義哪些工具可用、如何調用、需要什麼權限。包括：檔案系統訪問、shell 命令、API 調用、資料庫查詢、外部服務整合。

**好的工具編排**明確界定邊界：Agent 知道什麼可以做、什麼不能做、什麼需要人類先批准。

### 2. Guardrails 與安全約束

防止 Agent 執行有害行動的確定性規則：
- **權限邊界**：限制可訪問的檔案、目錄或命令
- **驗證檢查**：輸出應用前驗證（linter、類型檢查器、測試套件）
- **架構約束**：強制執行依賴邊界或命名約定等結構規則
- **速率限制**：防止無限迴圈或失控執行

### 3. 錯誤恢復與反饋迴圈

- 自動重試邏輯（遞增策略）
- 自我驗證迴圈（提交前檢查自己的工作）
- 回滾機制（變更破壞時恢復到之前狀態）
- 迴圈檢測（識別 Agent 是否卡住重複相同動作）

### 4. 可觀測性（Observability）

區分研究原型和生產系統的關鍵：
- 記錄每個 Agent 動作
- 追蹤 Token 使用量和成本
- 記錄決策點
- 浮現異常

**好問題**：為什麼 Agent 選擇這個工具？通過測試套件花了多少次嘗試？工作流中在哪裡花費了最多 Token？何時需要人類干預？

### 5. Human-in-the-Loop 檢查點

完全自主很少是合適的。設計何時以及如何諮詢人類：
- 明確批准門（「Agent 要執行 `rm -rf`，批准？」）
- 定期審查檢查點（人類評估較長任務的進展）

**目標**：不要人類微觀管理 Agent，而是在最高槓桿決策點放置人類判斷。

## 常見錯誤

| 錯誤 | 為什麼不行 | 應該怎麼做 |
|------|-----------|-----------|
| 一開始就設計完美 Harness | 猜不到 Agent 會犯什麼錯 | 從最小配置開始，出問題再加 |
| 裝一堆 MCP Server「以防萬一」 | 每個工具都消耗上下文預算 | 需要時再加 |
| CLAUDE.md 寫 500 行 | Agent 在冗長指令下表現更差 | 控制在 60 行，用 Skills 按需加載 |
| 每次改檔案都跑完整測試 | 5 分鐘的迴圈毀掉迭代速度 | 跑相關子集，完整套件放 CI |
| 只靠測試（傳感器） | Agent 反覆犯同樣的錯 | 加引導器預防 |
| 把約束寫在提示詞裡 | 提示詞是建議，lint 是強制 | 用計算型約束（lint、類型系統、Hooks）|
| 只靠引導器（無傳感器） | Agent 從不驗證結果 | 加入測試和類型檢查等傳感器 |

## 人類的角色

人類開發者帶來的是**隱性 Harness**：
- 吸收的約定和良好實踐
- 感受到的複雜性認知痛苦
- 知道自己的名字在 commit 上
- 攜帶組織對齊——意識到團隊試圖實現什麼、哪些技術債務因業務原因被容忍、「好」在這個特定上下文中的含義

Agent 沒有這些：沒有社會問責、沒有對 300 行函數的美學厭惡、沒有「我們這裡不這樣做」的直覺、沒有組織記憶。

**Harness 的目標**：不需要完全消除人類輸入，而是將其引導到人類輸入最重要的地方。
